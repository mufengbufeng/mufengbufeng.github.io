项目启动性能优化

背景

自我们项目上一次启动速度优化以来，最近又遇到了比较大的启动速度危机。上次优化主要从冗余的资源加载入手，本次则主要从代码逻辑上进行优化。

通过 log 信息，我们定位到启动时的加载逻辑主要有以下几个卡点：

DLL 编译 → config 解析 → 必要资源加载（场景、图集等） → SDK 登录

本文主要讲述 DLL、config 和 资源加载 三部分。

DLL 优化

DLL 部分基本都是引擎设置相关的内容，简单带过：

  • 删除 Unity 中非必要使用的包
  • 使用更高的剔除等级，配合更详细的 link.xml

表格数据加载逻辑

这是我们项目启动速度中最主要的卡点之一，单这部分加载时间就在 5 ~ 7 s 之间。

问题分析

主要原因是：在项目测试过程中，为了对比数据，我们增加了 A、B、C、D…… 等多个测试组。而由于我们是关卡制游戏，每个测试组的数据也在不断膨胀。再加上原始逻辑是串行加载，导致整体加载时间异常。

原始逻辑（串行）

修改逻辑（并行）

核心思路是：将串行逻辑改为并行，充分利用 CPU 性能。

资源提前加载

讲完表格的优化思路，资源加载的基本优化思路也是类似的：在加载阶段尽量让 CPU 全力工作。同时也顺带优化了一些经验性错误。

原有流程

DLL → 框架入口层（所有管理器初始化） → 加载资源（场景 / 图集） → 完成加载进入主界面 → 关闭 loading

这是大部分常规框架的思路，但其中存在一个问题：加载资源操作其实只需要在资源管理器初始化完成之后就可以开始，完全不需要等表格加载完毕、或音效管理器、场景管理器初始化完毕才开始加载。
